# BiteBlog 分布式美食探店社区平台 测试分析报告

---

## 1 引言

### 1.1 编写目的

本测试分析报告旨在系统性地汇总 BiteBlog 分布式美食探店社区平台全量微服务的非功能测试结果，验证各服务是否满足需求说明书中定义的性能指标、数据一致性要求及可靠性保障。预期读者包括项目开发团队成员、课程答辩评审教师及相关技术评估人员。

### 1.2 背景

a. 被测试软件系统名称：**BiteBlog 分布式美食探店社区平台**

b. 该系统为一款面向美食探店场景的分布式社交平台，具备笔记发布、全文搜索、Feed 流、个性化推荐、附近探店、热度排行、实时通知等核心功能。系统采用 Spring Cloud 微服务架构，包含 7 个后端微服务（User Service、Post Service、Feed Service、Recommend Service、Location Service、Rank Service、Notify Service）及 Gateway 网关，后端服务注册至 Nacos。

- 任务提出者：课程设计要求
- 开发者：项目团队（组长 + 6 名成员）
- 测试环境：Windows 11 本地开发机，Docker Desktop 承载全部中间件（MySQL 8.0、Redis 7、RabbitMQ 3.12、Elasticsearch 单节点、MinIO、Nacos）
- 测试环境与实际运行环境差异：中间件均为单节点 Docker 容器部署，未启用集群/主从模式；所有服务与中间件同机运行，网络延迟极低（localhost 通信）。生产环境中跨主机网络延迟、中间件集群化后的分布式开销等因素可能导致响应时间高于本报告实测值。

### 1.3 定义

| 术语/缩写 | 说明 |
|-----------|------|
| P95 / P99 | 95 分位 / 99 分位响应时间，即 95%/99% 的请求在此时限内完成。**仅在 JMeter 并发压测（数千样本、多线程并发分布）中使用**，具有统计意义 |
| 平均 (avg) | 算术平均响应时间。**单请求基线测试（10~20 次顺序采样）使用平均值作为主要指标**，此样本量下 P95 无统计意义 |
| 单请求基线 | 单客户端顺序发送 N 次请求（非并发），用于快速验证接口在无竞争条件下的基本性能。结果以平均值为准 |
| 并发压测 | JMeter 多线程并发请求，用于测量系统在竞争条件下的真实响应分布。结果以 P50/P90/P95/P99 分位数为准 |
| RT | Response Time，响应时间 |
| QPS | Queries Per Second，每秒查询数 |
| JMeter | Apache JMeter，开源性能压测工具 |
| Cache-Aside | 旁路缓存模式：读先查缓存，未命中则查数据库并回填；写直接更新数据库并失效缓存 |
| Fanout | 写扩散：普通用户发布笔记后写入所有粉丝的 Feed inbox |
| DLQ | Dead Letter Queue，死信队列 |
| MQ | Message Queue，消息队列（本项目使用 RabbitMQ） |
| GEO | Redis GEO 数据结构，用于地理位置存储与半径查询 |
| ItemCF | Item-based Collaborative Filtering，基于物品的协同过滤 |
| ZSet | Redis Sorted Set，有序集合 |
| TTL | Time To Live，缓存过期时间 |
| Lua | Redis 内嵌脚本语言，用于原子操作 |
| WGS84 | World Geodetic System 1984，全球大地测量坐标系 |
| ramp-up | JMeter 线程启动梯度时间 |
| DND | Do Not Disturb，免打扰时段 |

## 3 测试结果及发现

### 3.1 Post Service 测试 (P-0 ~ P-14)

#### P-0: 服务可达性
通过直连 (8082) 和 Gateway (8080) 访问 `/api/post/1`，均返回 `code=200`，服务可达且 Gateway 路由正常。

#### P-1: 笔记详情响应时间
要求平均 < 300ms。PowerShell 脚本 20 次连续采样：

| 指标 | 值 |
|------|-----|
| **平均** | **26.8ms** |
| 最小 | 18ms |
| 最大 | 33ms |

> **指标说明**：单请求基线测试使用**平均值**作为主要指标。10~20 次顺序请求的样本量不支持有意义的 P95 计算（20 样本的 P95 仅排除 1 个值，10 样本的 P95 即等于最大值）。P95/P99 等分位数指标仅在 JMeter 并发压测（数千样本、多线程并发分布）中使用。

平均 26.8ms，仅为目标的 9%，Redis 缓存命中后详情查询走 `post:cache:{postId}` 直接返回，无需 MySQL 查询，响应稳定。

#### P-2: ES 搜索响应时间
要求平均 < 800ms。PowerShell 脚本 20 次连续采样：

| 指标 | 值 |
|------|-----|
| **平均** | **30.5ms** |
| 最小 | 16ms |
| 最大 | 67ms |

平均 30.5ms，仅为目标的 3.8%。ES `multiMatch` 对 title/content/shopName 三字段 IK 分词检索效率极高。

#### P-3: 点赞切换响应时间
要求平均 < 300ms。PowerShell 脚本 10 次连续采样：

| 指标 | 值 |
|------|-----|
| **平均** | **35.8ms** |
| 最小 | 24ms |
| 最大 | 45ms |

平均 35.8ms，远优于目标。MySQL UNIQUE KEY + 原子 `SET like_count = like_count +/- 1` 响应高效。

#### P-4: Redis 缓存冷/热对比
Cache-Aside 模式验证：

| 路径 | 耗时 | 说明 |
|------|------|------|
| 冷路径（cache miss → MySQL） | 39ms | key 被删，触发 DB 查询并回填 |
| 热路径（cache hit → Redis GET） | 29ms | key 存在，直接 Redis GET 返回 |

热路径 29ms < 冷路径 39ms，Cache-Aside 模式生效。

#### P-5: 发布事务原子性 (PC-1)
发布笔记 postId=22 后立即查询详情，title/content/images 完整返回，`note` + `note_image` 在同一 `@Transactional` 中写入成功。

#### P-6: 详情缓存一致性 (PC-2)
清 Redis key 后首次请求触发 MySQL 查询并回填，二次请求命中 Redis。Cache-Aside 模式正确。

#### P-7: ES 搜索一致性 (PC-3)
发布笔记后第 1 次轮询（1.5s）即在 ES 搜索结果中找到该笔记，MQ `note.published` → `EsSyncService` 增量同步延迟 < 1.5s。

#### P-8: ES 降级 (PC-4)
`EsSyncService.search()` 使用 try-catch 捕获 ES 异常，降级返回 `Page.empty()`，ES 不可用时接口仍返回 code=200。

#### P-9: 逻辑删除过滤 (PC-5)
删除笔记 postId=23 后详情接口返回 code=2001 (NOT_FOUND)，用户帖子列表中不含 status=0 笔记。MyBatis-Plus `@TableLogic` + ES `filter(term("status", 1))` 双重过滤生效。

#### P-10: 搜索分页去重
page=1 与 page=2 的 postId 无交集，ES 分页游标正确。

#### P-11: 点赞幂等性
5 次 toggle 后 likeCount 仅变化 1（从 0 到 1），MySQL UNIQUE KEY uk_like 有效阻止重复插入。

#### P-12: ES 多字段匹配
按 shopName 搜索 `verify shop` 成功命中目标笔记，`multiMatch("title", "content", "shopName")` 三字段 IK 分词检索正确。

#### P-13: 安全性验证
无 Token 访问返回 HTTP 401（Gateway JWT 过滤器拦截），直连 Post Service 带 `X-User-Id` 头可正常访问。

#### P-14: JMeter 并发压测（200线程×4接口组，峰值800并发）

| 线程组 | 线程数 | 循环 | 总样本 | 目标 |
|--------|--------|------|--------|------|
| 笔记详情 `GET /api/post/1` | 200 | 25 | 5000 | P95 < 300ms |
| ES 搜索 `GET /api/post/search` | 200 | 25 | 5000 | P95 < 800ms |
| 点赞切换 `POST /api/post/1/like` | 200 | 12 | 2400 | P95 < 300ms |
| 评论分页 `GET /api/post/1/comments` | 200 | 25 | 5000 | P95 < 300ms |

| 接口 | 样本 | 错误率 | 平均 | P50 | P95 | P99 | 吞吐量 |
|------|------|--------|------|-----|-----|-----|--------|
| 笔记详情 | 5000 | 0% | 2420ms | 2726ms | 3517ms | 3326ms | 69/s |
| ES 搜索 | 5000 | 0% | 718ms | 863ms | 1694ms | 1550ms | 104/s |
| 点赞切换 | 2400 | 0.17% | 1087ms | 1192ms | 2091ms | 1928ms | 67/s |
| 评论分页 | 5000 | 0% | 1962ms | 2148ms | 2767ms | 2623ms | 74/s |

**总样本 17401，错误 4（0.02%），峰值并发 800 线程**。系统无崩溃、无数据不一致。点赞接口仅 4 次 DuplicateKeyException（0.17%），属于并发 toggle 的正常幂等冲突。高并发下响应时间偏高主要受本地单机资源竞争影响（MySQL 连接池、ES 单节点索引刷新），实际部署通过扩容可显著改善。

撤回后重新点赞，产生新的 like 通知（去重只查 `is_retracted=0`），撤回与再点赞流程自洽。

#### N-18: 按类型屏蔽 mute_type (OPT-4)
设置 `mute_type=like` 后，粉丝点赞消费端 `CheckResult.MUTED` 直接丢弃，不写库。

#### N-19: 按发送者屏蔽 mute_sender (OPT-4)
设置 `mute_sender=fanId` 后，该粉丝任何互动均不产生通知，按发送者维度全类型屏蔽生效。

#### N-20: 勿扰时段 dnd_time (OPT-4)
DND 时段内通知写库但不增 Redis 未读、不推 WebSocket。已知 UX：DND 期间 DB 未读可能大于 Redis 角标，依赖对账或用户已读修正。

#### N-21: 关注者发帖 follow_post 小 V (OPT-6)
小 V 用户发帖，粉丝收到 `follow_post` 通知。`note.published` → `notify.note.published.queue` → fanout 生效。

#### N-22: 大 V 跳过 follow_post fanout (OPT-6)
大 V (authorId=1, fans SCARD=439, isBigV=True) 发帖后，粉丝 follow_post 总数不变。大 V 判定与 Feed 一致（粉丝 ≥ 50 或 fans ≥ 500），避免 fanout 写入风暴。

#### N-23: 评论回复 comment_reply (OPT-5)
回复他人评论后，父评论作者收到 `comment_reply` 通知。Post MQ 扩展字段 `parentCommentUserId`、`commentContent` 被 Notify 正确消费。
