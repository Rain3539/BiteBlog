# BiteBlog 接口说明文档

## 1. 统一响应格式

所有接口返回统一 JSON 格式：

```json
{
  "code": 200,
  "msg": "success",
  "data": { ... },
  "timestamp": "2026-05-08T15:42:19+08:00",
  "requestId": "trace-uuid"
}
```

### 错误码

| Code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 401 | 未登录或 Token 过期 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 429 | 请求频率超限 |
| 500 | 系统内部错误 |
| 1001 | 用户不存在 |
| 1002 | 手机号已注册 |
| 1003 | 密码错误 |
| 1004 | 账号已被禁用 |
| 2001 | 笔记不存在或已删除 |
| 2002 | 图片上传失败 |
| 2003 | 笔记内容不合法 |
| 3001 | Feed 无新内容 |
| 4001 | 推荐服务暂不可用 |
| 5001 | 地理位置获取失败 |
| 6001 | 通知发送失败 |

## 2. 鉴权方式

- 登录成功后返回 JWT Token
- 需要鉴权的接口在请求头携带：`Authorization: Bearer <token>`
- Gateway 统一校验 Token，将解析后的 `userId` 通过 `X-User-Id` 请求头传递给下游服务
- 白名单接口（无需 Token）：`/user/login`、`/user/register`、`/rank/top10`

## 3. 分页约定

### 游标分页（Feed / 推荐）

```
GET /api/feed/timeline?cursor=1714972800000&size=20
```

响应：

```json
{
  "code": 200,
  "data": {
    "list": [...],
    "cursor": "1714972700000",
    "hasMore": true
  }
}
```

### 传统分页（评论 / 排行 / 通知）

```
GET /api/post/1/comments?page=1&size=20
```

响应：

```json
{
  "code": 200,
  "data": {
    "list": [...],
    "total": 156
  }
}
```

## 4. 网关路由规则

```
/api/user/**       → user-service     (8081)
/api/post/**       → post-service     (8082)
/api/feed/**       → feed-service     (8083)
/api/recommend/**  → recommend-service (8084)
/api/location/**   → location-service (8085)
/api/rank/**       → rank-service     (8086)
/api/notify/**     → notify-service   (8087)
```

Gateway 自动 StripPrefix `/api`，下游服务看到的是 `/user/**`、`/post/**` 等。

#### 热度分计算规则

当前 Rank Service 使用 Redis Sorted Set 维护热度排行榜，按照 `hotScore` 从高到低排序。热度分计算公式如下：

```text
hotScore = likeCount × 3
         + collectCount × 5
         + commentCount × 4
         + (scoreTaste + scoreSmell + scoreColor) / 3
         + 24 / sqrt(hoursSinceCreated)
```

其中：

- `likeCount`：点赞数，权重为 3；
- `collectCount`：收藏数，权重为 5；
- `commentCount`：评论数，权重为 4；
- `scoreTaste`、`scoreSmell`、`scoreColor`：分别表示口味、气味、色泽评分；
- `24 / sqrt(hoursSinceCreated)`：时间新鲜度加成，发布时间越近，加成越高。

例如测试数据中：

```text
热榜测试笔记3：
likeCount = 45
collectCount = 20
commentCount = 10
scoreTaste = 5
scoreSmell = 4
scoreColor = 5
```

其热度分约为：

```text
45×3 + 20×5 + 10×4 + (5+4+5)/3 + 24
= 303.6667
```

因此该笔记在测试排行榜中排名第一。

#### Redis Sorted Set 实现

排行榜使用 Redis Sorted Set 存储，key 设计如下：

```text
rank:hot:daily
rank:hot:weekly
rank:hot:all
```

其中：

- member：笔记 ID；
- score：热度分 `hotScore`；
- 查询时使用倒序排序，分数越高排名越靠前。

#### 定时刷新机制

Rank Service 已实现定时刷新机制，通过 `@Scheduled` 定时任务每 10 分钟重建一次排行榜缓存，避免缓存长期不一致。

```java
@Scheduled(cron = "0 0/10 * * * ?")
public void scheduledRefresh() {
    for (String type : TYPES) {
        rebuild(type);
    }
}
```

#### RabbitMQ 事件加分机制

Rank Service 已实现 RabbitMQ 事件监听逻辑，用于在用户产生互动行为时增量更新排行榜分数。

事件类型及加分规则如下：

| 事件类型 | 说明 | 分数增量 |
| -------- | ---- | -------- |
| like     | 点赞 | +3       |
| collect  | 收藏 | +5       |
| comment  | 评论 | +4       |
| view     | 浏览 | +1       |

对应逻辑：

- 发布笔记事件：计算初始热度分并加入榜单；
- 删除笔记事件：从 `daily`、`weekly`、`all` 三类榜单中移除；
- 点赞、收藏、评论、浏览事件：对 Redis Sorted Set 中对应笔记进行增量加分。

本地测试时 RabbitMQ 监听可临时关闭，避免本机未启动 RabbitMQ 导致连接失败；完整联调时需要启动 RabbitMQ，并开启监听器。

#### 非功能需求说明

排行服务当前满足以下非功能需求：

| 类型     | 说明                                                         |
| -------- | ------------------------------------------------------------ |
| 性能     | 使用 Redis Sorted Set 存储排行榜，查询 Top10 和分页排行时避免直接扫描 MySQL，提高读取性能。 |
| 可用性   | Redis 缓存为空时，可通过 `/rank/rebuild` 手动重建；同时支持定时任务周期性刷新。 |
| 可扩展性 | 支持 `daily`、`weekly`、`all` 多榜单类型，后续可扩展城市榜、分类榜、附近榜等。 |
| 一致性   | 通过定时全量重建 + RabbitMQ 增量加分结合，兼顾实时性和最终一致性。 |
| 可测试性 | 已通过 curl 对健康检查、排行榜重建、Top10 查询、分页查询进行本地验证。 |
| 容错性   | 本地测试可关闭 RabbitMQ Listener，避免依赖组件未启动导致主服务不可用。 |
