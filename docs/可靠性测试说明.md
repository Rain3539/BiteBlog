# BiteBlog 可靠性测试说明

## 1. 概述

可靠性目标：**出现问题后数据不丢、不乱、不重复**。各服务通过不同机制保障这一目标：事务边界、消息持久化、死信队列、重试、幂等去重、降级 fallback、定时自愈。

---

## 2. Post Service — 事件发布可靠性

### 2.1 afterCommit：事务回滚不发 MQ

**机制**：`PostEventPublisher` 在 Spring 事务 `afterCommit` 回调中才发送 MQ。MySQL 回滚 → MQ 不发出，杜绝脏事件。

| 测试场景 | 预期 |
|----------|------|
| 发布笔记时 MySQL 插入失败回滚 | `note.published` 不发出，下游无感知 |
| 删除笔记时 MySQL 更新失败回滚 | `note.deleted` 不发出 |

### 2.2 消息持久化：Broker 重启不丢

**机制**：Exchange/Queue 声明为 `durable`，消息投递模式 `PERSISTENT`。

| 测试场景 | 预期 |
|----------|------|
| 发布笔记后立即重启 RabbitMQ | 重启后消息仍在队列，消费者恢复后正常消费 |

## 3. Notify Service — 消息消费可靠性 + 幂等 + 降级自愈

Notify 承担「MQ 异步写入 → MySQL 持久化 → Redis 未读缓存 → WebSocket 推送」链路。可靠性重点：**消费失败不丢消息、重投不重复写、Redis 异常仍可查、长期运行缓存可自愈**。

**对应测试编号**：`NR-1`（手动 Ack + DLQ）、`NR-2`（5 分钟幂等去重）；脚本验证见 `F-5`、`F-6`、`F-9`（NC-8/NC-10）、`F-12`/`F-13`。执行：`测试脚本/notify-test-verify.ps1`。

### 3.1 手动 Ack + 死信队列（DLQ）

**机制**：`NotifyEventListener` 对 `notify.interaction.queue` 与 `notify.note.published.queue` 均配置 `ackMode = "MANUAL"`。消费成功 `basicAck`；业务异常 `basicNack(requeue=false)` → 经 `notify.dlx` 路由到 `notify.dead.queue`。消息不静默丢失，运维可检查死信队列后人工重放。

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| 消费时抛 RuntimeException | 消息进 `notify.dead.queue`，主队列不无限堆积 | **已验证：DLQ ready=0，主队列 unacked=0**（`F-5` / `NR-1`） |
| 重启 Notify 服务 | 未 Ack 消息重新投递，恢复后继续消费 | 代码路径覆盖；RabbitMQ consumers=1 |
| `note.published` 消费失败 | 与 interaction 队列相同 DLX 链路 | 队列拓扑已在 `NotifyRabbitConfig` 声明 |

**RabbitMQ 实测指标**（`notify-test-verify.ps1` §16）：

| 队列 | Consumers | Ready | Unacked |
|------|-----------|-------|---------|
| `notify.interaction.queue` | 1 | 0 | 0 |
| `notify.note.published.queue` | 1 | — | — |
| `notify.dead.queue` | — | 0 | — |

### 3.2 幂等去重（5 分钟窗口）

**机制**：写入前查询 5 分钟内是否已存在相同 `(receiver_id, sender_id, type, biz_id)` 且 `is_retracted=0` 的记录。覆盖 MQ 重投（秒级~分钟级），又**不误伤**「取消点赞 → 再点赞」：撤回后 `is_retracted=1`，去重查不到旧记录，可正常产生新通知。

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| 5 分钟内 MQ 重复投递同一 interaction.add | 仅 1 条通知，第二条 `dedup skip` | **已验证**（`F-6` / `NC-5` / `NR-2`） |
| 取消点赞后再点赞 | 撤回旧通知 + 产生新通知，总数正确 | **已验证**（`F-12` / `F-13`） |

### 3.3 事务边界：afterCommit 再调远程

**机制**：`handleInteractionEvent` 标注 `@Transactional`；DB insert 在事务内完成；Feign 查昵称与 WebSocket 推送通过 `TransactionSynchronizationManager.afterCommit` 延迟到事务提交后执行。避免远程 I/O 占用 DB 连接，且 **MySQL 回滚时不推送脏通知**。

| 测试场景 | 预期 |
|----------|------|
| 写库事务失败回滚 | 不 INCR Redis 未读、不推 WS |
| Feign / WS 失败 | 仅 warn 日志，不影响已提交的通知记录 |

### 3.4 Redis 未读数降级（NC-10）

**机制**：`countUnread()` 优先读 Redis `notify:unread:{userId}`（`StringRedisTemplate` 纯整数字符串）；Redis 异常时 catch 后降级 `SELECT COUNT(*) WHERE read_status=0 AND is_retracted=0`，接口仍返回 200。

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| `docker pause` Redis 容器后请求 unread-count / list | code=200，未读数来自 MySQL | **已验证**（`F-9` / `NC-10`） |
| Redis 恢复后 | cache miss 回填，后续恢复热路径 | 代码路径覆盖 |

### 3.5 未读缓存精准维护 + 定时对账（OPT-2）

**机制**：

- 写通知：`INCR`（勿扰 DND 除外，见 3.7）
- 单条已读：`DECR`（非 DEL key，避免高频 DB COUNT）
- 全部已读：`SET "0"`（非 DEL，角标立即归零）
- 取消互动撤回：未读通知 `DECR`
- **定时自愈**：`reconcileUnreadCache()` 每 10 分钟扫描已有 `notify:unread:*` key，与 DB COUNT 对比，漂移时用 DB 值覆盖 Redis

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| 单条已读后 unread-count | Redis DECR，与 list 未读 total 一致 | **已验证**（`NC-3`） |
| read-all 后 unread-count | Redis SET `"0"`，API 返回 0 | **已验证**（`NC-4`） |
| DND 写库不增 Redis，长期漂移 | 10 分钟对账或用户已读后修正 | 设计如此；脚本登录后 read-all 重置基线 |

### 3.6 冷热归档幂等（NC-9）

**机制**：`archiveOldReadNotifications()` 每天凌晨 3 点执行；仅迁移 30 天前**已读**记录；每批 500 条；冷表 `INSERT IGNORE` 保证任务重跑幂等；**未读通知不归档**。

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| 定时任务重复执行 | 不重复插入归档表、不丢数据 | 代码审查 + cron 声明（`NC-9`） |
| 未读超 30 天通知 | 仍保留在热表 | 设计约束 |

### 3.7 偏好与勿扰（OPT-4）对可靠性的影响

**机制**：`NotifyPreferenceService.check()` 返回 `MUTED` / `DND` / `ALLOW`：

| 结果 | 写库 | Redis 未读 | WS 推送 |
|------|------|------------|---------|
| MUTED | 否 | 否 | 否 |
| DND | 是（read_status=0） | 否 | 否 |
| ALLOW | 是 | INCR | 是 |

**可靠性说明**：DND 故意造成「DB 未读 > Redis 角标」的短期不一致，通知不丢失（仍可在列表查看），依赖 10 分钟对账或用户已读操作修正角标。`F-20` 已验证 DND 写库但不增未读。

### 3.8 自操作过滤与多队列消费

**机制**：

- `authorId == userId` 时直接 ack，不写库（`NC-6` / `F-7`）
- 双队列独立监听：`interaction.*`（点赞/收藏/评论/撤回）与 `note.published`（follow_post fanout）；互不影响，各自 DLQ

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| 作者给自己笔记点赞 | notification 表不新增 | **已验证**（`F-7`） |
| Post 发布笔记 | follow_post 向粉丝 fanout（小 V）；大 V 跳过防风暴 | **已验证**（`F-21` / `F-22`） |

## 4. Feed Service — Redis 不可用时降级

**机制**：inbox 为空或过滤后无结果时，自动降级到直接查 MySQL（`getTimelineFromDb`），并回填 Redis 预热。

| 测试场景 | 预期 |
|----------|------|
| 清空 Redis inbox 后请求 feed | 降级到 MySQL 查询，正常返回 |
| 降级后再请求 | 首次已回填 inbox，后续命中 Redis |

## 5. Rank Service — 缓存自愈

### 5.1 缓存为空时自动重建

**机制**：`ensureCache()` 检测 Redis ZSet 为空时立即触发 `rebuild()` 从 MySQL 全量重建。

| 测试场景 | 预期 |
|----------|------|
| 手动删除 `rank:daily:{date}` 后请求榜单 | 自动触发 rebuild，从 MySQL 重建 |

### 5.2 定时重建（每 10 分钟）

**机制**：`@Scheduled(cron = "0 0/10 * * * ?")` 全量重建日榜、周榜、总榜，兜底修复任何 Redis 数据异常。

| 测试场景 | 预期 |
|----------|------|
| 手动篡改 Redis rank score，等待 10 分钟 | score 被修复为正确值 |
| 手动添加过期笔记到日榜，等待 10 分钟 | 过期笔记被清除 |

## 6. Location Service — 跨服务事务时序重试 + 定时自愈

### 6.1 消费重试

**机制**：`addNoteLocation()` 在查不到 note 时最多重试 5 次（间隔 200ms），等待 Post Service 事务提交完成。

| 测试场景 | 预期 | 验证结果 |
|----------|------|----------|
| Post 刚提交事务，Location 立即消费 MQ | 重试后查到 note 并写入 GEO | **已验证：500ms 内 GEO 写入确认** |
| Post 事务回滚，note 不存在 | 重试 5 次后跳过，日志 warn | **已验证：代码 warn 日志路径覆盖** |

### 6.2 定时自愈（每 10 分钟）

**机制**：`@Scheduled(cron = "0 0/10 * * * ?")` 每 10 分钟从 MySQL 全量重建 `location:notes`。遍历所有 `status=1 AND longitude IS NOT NULL AND latitude IS NOT NULL` 的笔记 → 删除旧 key → GEOADD 批量写入。兜底修复 MQ 消息丢失、Redis 重启、GEO 数据漂移等异常。

| 测试场景 | 预期 |
|----------|------|
| 手动删除 `location:notes` key 后等待 10 分钟 | 定时任务从 MySQL 全量重建 GEO |
| MQ 消息丢失导致个别笔记未写入 GEO | 下一次定时任务自动补齐 |

**对应测试编号**：`L-8`（交叉功能验证）、`L-11`（定时重建自愈）、`L-12`（MQ 队列诊断）。
测试结果：`location.note.published.queue` 和 `location.note.deleted.queue` 均 ready=0、unacked=0、consumers=1，无消息积压。交叉功能链路中发布后首次轮询（500ms）即确认 GEO 写入。定时重建已在服务代码中落地。

## 7. Recommend Service — 多级降级链

### 7.1 冷启动降级：两级 fallback

**机制**：`rank:daily:{date}` → MySQL hot notes。Rank 日榜为空或 Redis 不可用时自动 fall 到 MySQL 热门笔记。

| 测试场景 | 预期 |
|----------|------|
| 清空 `rank:daily:{date}` 后请求推荐 | 降到 MySQL 热门笔记，正常返回 |
| 同时停 ES + 清空 Redis | 降到 MySQL，正常返回 |

### 7.2 ItemCF 召回降级：两级 fallback

**机制**：Redis 预计算 `recommend:itemcf:similar:{postId}` → MySQL 在线行为扫描。这条链路只有两级；如果 ItemCF 两级都没有足够候选，外层推荐流程再走 Rank 日榜 / MySQL 热门补量。

| 测试场景 | 预期 |
|----------|------|
| 清空 Redis ItemCF 相似池后请求个性化推荐 | ItemCF 走 MySQL 在线行为扫描 fallback |

### 7.3 曝光预占降级：Lua → 本地过滤

**机制**：Redis Lua 原子预占为主路径，Redis 异常时降级为本地 Set 过滤。

| 测试场景 | 预期 |
|----------|------|
| 停止 Redis 后请求推荐 | 降级到本地曝光过滤，不报错 |

### 7.4 定时预计算自愈（每 10 分钟）

**机制**：`@Scheduled(initialDelay=60s, fixedDelay=10min)` 刷新 Rank 日榜和 Redis ItemCF 相似池。

| 测试场景 | 预期 |
|----------|------|
| 手动清空 `rank:daily:{date}`，等待 10 分钟 | 预计算任务从 MySQL 重建 |
| 手动删除 Redis `recommend:itemcf:similar:*`，等待 10 分钟 | 预计算任务重建相似池 |

## 8. 各服务可靠性责任汇总

| 服务 | 核心机制 | 保障目标 |
|------|----------|----------|
| **Post** | afterCommit 发送、消息 PERSISTENT | DB 回滚不发脏事件；Broker 重启消息不丢 |
| **Notify** | 手动 Ack + DLQ（双队列）、5 分钟去重、afterCommit 远程调用、Redis 未读降级、10 分钟对账、归档 INSERT IGNORE、is_retracted 软撤回 | 消费失败不丢消息；重投不重复；Redis 宕机可查；长期缓存漂移可自愈 |
| **Feed** | Redis → MySQL 降级 | Redis 宕机仍可用 |
| **Rank** | ensureCache 自动重建 + 10 分钟定时重建 | Redis 数据异常时自动从 MySQL 修复 |
| **Location** | 5 次重试等 Post 事务提交 | 跨服务事务时序差自动补偿 |
| **Recommend** | 多级 fallback 链 + 10 分钟定时预计算 | ES/Redis/部分远程作者信息不可用时仍尽量返回结果 |

## 9. 已知可靠性差距

| 问题 | 影响 | 涉及服务 |
|------|------|----------|
| Feed/Rank 消费端未配置 DLQ（仅 Notify 有） | AUTO ack + 重试耗尽后消息丢失 | Feed、Rank |
| Feed fanout 逐粉丝 ZADD，非事务 | Redis 中途断连导致部分粉丝收不到 | Feed |
| Notify DND 写库不增 Redis 未读 | 勿扰期间 DB 未读与角标短期不一致，依赖对账/已读修正 | Notify |
| Post 评论删除无 interaction.remove 事件 | 评论通知无法软撤回 | Post / Notify |
| Feign 调用无断路器 | Post/User 不可用时拖慢 Recommend 响应 | Recommend |
