# BiteBlog 数据库说明文档

## 概述

BiteBlog 使用 **MySQL 8.0** 作为主存储，共 **12 张表**，覆盖用户、笔记、社交互动、行为记录、通知及通知偏好等业务场景。数据库名 `biteblog`，字符集 `utf8mb4`，排序规则 `utf8mb4_unicode_ci`。

数据初始化通过 `sql/init.sql` 在 Docker 容器首次启动时自动执行（`docker-entrypoint-initdb.d`）。

**逻辑删除**：user 表和 note 表使用 `status` 字段做逻辑删除（0=删除/禁用，1=正常），由 MyBatis-Plus `@TableLogic` 自动处理。

## 表结构说明

### 1. 用户表 (user)

存储平台注册用户的基本信息和统计数据。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 用户 ID |
| phone | VARCHAR(20) | NOT NULL, UNIQUE | 手机号（登录账号，唯一） |
| username | VARCHAR(50) | NOT NULL, UNIQUE | 用户名（唯一） |
| password_hash | VARCHAR(128) | NOT NULL | 密码哈希值（BCrypt 加密） |
| avatar | VARCHAR(512) | DEFAULT NULL | 头像 URL（MinIO 地址） |
| bio | VARCHAR(200) | DEFAULT NULL | 个人简介 |
| follower_count | INT | NOT NULL, DEFAULT 0 | 粉丝数（冗余字段，避免 COUNT 查询） |
| following_count | INT | NOT NULL, DEFAULT 0 | 关注数（冗余字段） |
| like_count | INT | NOT NULL, DEFAULT 0 | 获赞总数（冗余字段） |
| is_big_v | TINYINT(1) | NOT NULL, DEFAULT 0 | 是否大 V（0=否，1=是，粉丝 >= 5000 触发） |
| status | TINYINT(1) | NOT NULL, DEFAULT 1 | 逻辑删除状态（0=禁用/删除，1=正常） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 注册时间 |
| updated_at | DATETIME | ON UPDATE CURRENT_TIMESTAMP | 最后更新时间 |

**索引：**
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_phone (phone)` — 手机号唯一，防止重复注册
- `UNIQUE KEY uk_username (username)` — 用户名唯一

**关联关系：**
- 被 `follow_relation.user_id` / `follow_relation.target_user_id` 引用
- 被 `note.author_id` 引用
- 被 `note_like.user_id`、`note_favorite.user_id`、`comment.user_id` 引用
- 被 `user_behavior.user_id` 引用
- 被 `notification.receiver_id` / `notification.sender_id` 引用
- 被 `notification_preference.user_id` 引用

**使用场景：**
- 注册时检查手机号/用户名唯一性
- 登录时根据手机号查询并验证 BCrypt 密码
- 关注/取关时更新 `follower_count` 和 `following_count`
- 判断是否大 V（`is_big_v`）以决定 Feed 分发策略（推 or 拉）

### 2. 关注关系表 (follow_relation)

存储用户之间的关注关系。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 自增主键 |
| user_id | BIGINT | NOT NULL | 关注者 ID（谁关注） |
| target_user_id | BIGINT | NOT NULL | 被关注者 ID（关注谁） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 关注时间 |

**索引：**
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_follow (user_id, target_user_id)` — 同一用户不能重复关注同一目标
- `INDEX idx_target (target_user_id)` — 加速粉丝列表查询

**关联关系：**
- `user_id` → `user.id`（关注者）
- `target_user_id` → `user.id`（被关注者）

**使用场景：**
- 关注/取关操作：插入或删除记录，同时更新 user 表计数
- 查询关注列表：`WHERE user_id = ?`，分页
- 查询粉丝列表：`WHERE target_user_id = ?`，分页
- Feed 分发：查询发布者的所有粉丝 ID，fanout 到粉丝 inbox

### 3. 探店笔记表 (note)

存储探店笔记的核心内容。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 笔记 ID |
| author_id | BIGINT | NOT NULL | 作者用户 ID |
| title | VARCHAR(100) | NOT NULL | 笔记标题 |
| content | TEXT | NOT NULL | 笔记正文内容 |
| shop_name | VARCHAR(100) | DEFAULT NULL | 探店店铺名称 |
| address | VARCHAR(200) | DEFAULT NULL | 店铺地址 |
| longitude | DECIMAL(10,7) | DEFAULT NULL | 经度（WGS84 坐标） |
| latitude | DECIMAL(10,7) | DEFAULT NULL | 纬度（WGS84 坐标） |
| score_color | TINYINT | DEFAULT 0 | 环境评分（1-5 分） |
| score_smell | TINYINT | DEFAULT 0 | 卫生评分（1-5 分） |
| score_taste | TINYINT | DEFAULT 0 | 口味评分（1-5 分） |
| like_count | INT | NOT NULL, DEFAULT 0 | 点赞数（冗余字段） |
| collect_count | INT | NOT NULL, DEFAULT 0 | 收藏数（冗余字段） |
| comment_count | INT | NOT NULL, DEFAULT 0 | 评论数（冗余字段） |
| status | TINYINT | NOT NULL, DEFAULT 1 | 状态（0=删除，1=正常，2=审核中） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 发布时间 |
| updated_at | DATETIME | ON UPDATE CURRENT_TIMESTAMP | 最后更新时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_author (author_id)` — 加速查询某用户的所有笔记
- `INDEX idx_created (created_at)` — 加速按时间排序

**关联关系：**
- `author_id` → `user.id`
- 被 `note_image.note_id`、`note_like.note_id`、`note_favorite.note_id`、`comment.note_id` 引用

**使用场景：**
- 发布笔记：写入 note 表 + note_image 表，同步 ES post_index，发送 RabbitMQ 事件
- 笔记详情：Redis 缓存优先，未命中回源 MySQL
- 点赞/收藏/评论：更新对应的 `like_count` / `collect_count` / `comment_count`
- 附近探店：使用 `longitude` / `latitude` 写入 Redis GEO

### 4. 笔记图片表 (note_image)

存储笔记关联的图片 URL（图片文件存储在 MinIO）。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 自增主键 |
| note_id | BIGINT | NOT NULL | 所属笔记 ID |
| image_url | VARCHAR(512) | NOT NULL | 图片 MinIO 访问 URL |
| sort_order | TINYINT | NOT NULL, DEFAULT 0 | 排序序号（封面图 = 0） |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_note (note_id)` — 加速查询某笔记的所有图片

**关联关系：**
- `note_id` → `note.id`

**使用场景：**
- 发布笔记时上传图片到 MinIO，将返回的 URL 写入此表
- 查询笔记详情时 JOIN 此表获取图片列表
- `sort_order = 0` 的图片作为封面缩略图展示

### 5. 点赞表 (note_like)

记录用户对笔记的点赞关系。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 自增主键 |
| note_id | BIGINT | NOT NULL | 笔记 ID |
| user_id | BIGINT | NOT NULL | 点赞用户 ID |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 点赞时间 |

**索引：**
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_like (note_id, user_id)` — 同一用户对同一笔记只能点赞一次（幂等）
- `INDEX idx_user (user_id)` — 加速查询某用户点赞过的笔记

**关联关系：**
- `note_id` → `note.id`
- `user_id` → `user.id`

**使用场景：**
- 点赞：尝试 INSERT，唯一约束保证幂等（重复点赞不报错，返回已赞状态）
- 取消点赞：DELETE 记录
- 点赞时同步更新 `note.like_count`，发送 RabbitMQ 事件通知 Rank Service 加分

---

### 6. 收藏表 (note_favorite)

记录用户对笔记的收藏关系，结构与 note_like 完全一致。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 自增主键 |
| note_id | BIGINT | NOT NULL | 笔记 ID |
| user_id | BIGINT | NOT NULL | 收藏用户 ID |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 收藏时间 |

**索引：**
- `PRIMARY KEY (id)`
- `UNIQUE KEY uk_favorite (note_id, user_id)` — 同一用户对同一笔记只能收藏一次
- `INDEX idx_user (user_id)`

**使用场景：**
- 与 note_like 相同的幂等机制
- 收藏权重高于点赞。当前 Rank Service 热度分中，点赞权重为 3，收藏权重为 5，评论权重为 4。

### 7. 评论表 (comment)

存储笔记下的评论，支持二级回复（parent_id）。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 评论 ID |
| note_id | BIGINT | NOT NULL | 所属笔记 ID |
| user_id | BIGINT | NOT NULL | 评论者 ID |
| parent_id | BIGINT | DEFAULT NULL | 父评论 ID（NULL=顶级评论，非 NULL=回复） |
| content | VARCHAR(500) | NOT NULL | 评论内容 |
| status | TINYINT | NOT NULL, DEFAULT 1 | 状态（0=删除，1=正常） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 评论时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_note (note_id)` — 加速查询某笔记的评论
- `INDEX idx_parent (parent_id)` — 加速查询某评论的回复

**关联关系：**
- `note_id` → `note.id`
- `user_id` → `user.id`
- `parent_id` → `comment.id`（自关联，支持二级回复）

**使用场景：**
- 发表评论：INSERT + 更新 `note.comment_count`
- 评论列表：`WHERE note_id = ? AND parent_id IS NULL` 查顶级评论，传统分页
- 回复：`WHERE parent_id = ?` 查某条评论的回复

---

### 8. 用户行为表 (user_behavior)

记录用户对笔记的行为（浏览、停留、点赞、收藏、评论），用于推荐算法。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 自增主键 |
| user_id | BIGINT | NOT NULL | 用户 ID |
| note_id | BIGINT | NOT NULL | 笔记 ID |
| behavior_type | VARCHAR(20) | NOT NULL | 行为类型（view/dwell/like/collect/comment） |
| weight | INT | NOT NULL, DEFAULT 1 | 行为权重（不同行为有不同权重） |
| dwell_time | INT | DEFAULT NULL | 停留时长（秒，仅 behavior_type=dwell 时有值） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 行为发生时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_user_type (user_id, behavior_type)` — 加速查询某用户的特定行为
- `INDEX idx_note (note_id)` — 加速查询某笔记的所有行为

**关联关系：**
- `user_id` → `user.id`
- `note_id` → `note.id`

**使用场景：**
- 推荐算法：根据用户的 like/collect/comment 行为构建兴趣画像
- ItemCF 协同过滤：通过 MySQL 行为数据预计算相似度，并写入 Redis ZSet
- 曝光去重：用户浏览过的笔记加入 `exposure:{userId}` Redis Set

**行为权重参考：**

| behavior_type | weight | 含义 |
|---------------|--------|------|
| view | 1 | 浏览（滑过 Feed 流） |
| dwell | 3 | 停留（打开笔记详情） |
| like | 5 | 点赞 |
| collect | 8 | 收藏 |
| comment | 10 | 评论 |

---

### 9. 通知表 (notification)

存储系统通知（点赞、收藏、评论、回复、关注者发帖等触发）。Notify Service 热表，仅保留近 30 天活跃数据；过期已读记录迁入 `notification_archive`。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 通知 ID |
| receiver_id | BIGINT | NOT NULL | 接收者用户 ID |
| sender_id | BIGINT | NOT NULL | 触发者用户 ID |
| type | VARCHAR(20) | NOT NULL | 通知类型：`like` / `collect` / `comment` / `comment_reply` / `follow_post` |
| biz_id | BIGINT | DEFAULT NULL | 关联业务 ID（通常为笔记 ID） |
| content | VARCHAR(200) | DEFAULT NULL | 通知摘要文本 |
| read_status | TINYINT | NOT NULL, DEFAULT 0 | 已读状态（0=未读，1=已读） |
| is_retracted | TINYINT(1) | NOT NULL, DEFAULT 0 | 是否已撤回（0=正常，1=已撤回；取消点赞/收藏时软删除，列表不可见） |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 通知创建时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_receiver (receiver_id, read_status, is_retracted)` — 覆盖列表/未读 COUNT 最常见查询
- `INDEX idx_created (created_at)` — 加速按时间排序与归档任务

**关联关系：**
- `receiver_id` → `user.id`（通知接收者）
- `sender_id` → `user.id`（触发通知的用户）

**使用场景：**
- Post 互动 / 发帖事件经 RabbitMQ 由 Notify Service 消费后写入
- 列表查询：`is_retracted = 0`，排序 `created_at DESC, id DESC`
- 未读计数：`read_status = 0 AND is_retracted = 0`；Redis `notify:unread:{userId}` 缓存（StringRedisTemplate 纯整数字符串）
- 取消点赞/收藏：`is_retracted = 1`，若原未读则 Redis DECR
- 勿扰时段（dnd）：仍写库但不增 Redis 未读、不推 WebSocket
- 自己给自己互动不产生通知（自操作过滤）

---

### 10. 通知归档表 (notification_archive)

存储 30 天前**已读**通知的冷数据，由 Notify Service 定时任务迁移，减轻热表压力。字段结构与热表一致（不含 `is_retracted`，归档时均为已读记录）。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK | 来自热表的原始 ID（非自增） |
| receiver_id | BIGINT | NOT NULL | 接收者用户 ID |
| sender_id | BIGINT | NOT NULL | 发送者用户 ID |
| type | VARCHAR(20) | NOT NULL | 通知类型：`like` / `collect` / `comment` / `comment_reply` / `follow_post` |
| biz_id | BIGINT | DEFAULT NULL | 关联业务 ID（通常为笔记 ID） |
| content | VARCHAR(200) | DEFAULT NULL | 通知摘要 |
| read_status | TINYINT | NOT NULL, DEFAULT 1 | 归档时均为已读 |
| created_at | DATETIME | NOT NULL | 原始创建时间 |
| archived_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 归档时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_archive_receiver (receiver_id, created_at)` — 按用户查历史归档

**使用场景：**
- 每天凌晨 3 点批量迁移：`INSERT IGNORE` 幂等，先写冷表再删热表
- **未读通知不归档**，保证用户历史未读不丢失

### 11. 通知偏好表 (notification_preference)

存储用户对通知的个性化设置（类型屏蔽、发送者屏蔽、勿扰时段）。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 偏好记录 ID |
| user_id | BIGINT | NOT NULL | 设置者（通知接收方） |
| pref_type | VARCHAR(30) | NOT NULL | `mute_type` / `mute_sender` / `dnd_time` |
| pref_value | VARCHAR(100) | NOT NULL | 类型名 / senderId / `HH:mm-HH:mm` 时段 |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_user_type (user_id, pref_type)` — 按用户加载偏好

**pref_type 说明：**

| pref_type | pref_value 示例 | 效果 |
|-----------|------------------|------|
| mute_type | `like` / `collect` / `comment` / `follow_post` | 消费端丢弃该类型通知；屏蔽 `comment` 时同时屏蔽 `comment_reply` |
| mute_sender | `4`（userId） | 丢弃来自该发送者的所有通知 |
| dnd_time | `22:00-08:00` | 写库但不增 Redis 未读、不推 WebSocket |

**使用场景：**
- Notify Service `NotifyPreferenceService.check()` 在 MQ 消费写库前判定 ALLOW / MUTED / DND
- 偏好缓存 Redis `notify:pref:{userId}`（JSON，TTL 10 分钟）
- 前端 `NotifyPreferenceDrawer` 提供设置 UI

### Notify 服务 MQ 事件与落库映射

Notify Service 不直接写 Post/User 业务表，仅消费 MQ 事件写入 `notification`，并读取 Redis 辅助 fanout 与偏好判定：

| 事件来源 | 交换机 | 路由键 | 队列 | 落库 type | 说明 |
|----------|--------|--------|------|-----------|------|
| 点赞/取消 | `biteblog.interaction` | `interaction.like` | `notify.interaction.queue` | `like` | `action=add` 写库；`action=remove` 软撤回 `is_retracted=1` |
| 收藏/取消 | `biteblog.interaction` | `interaction.collect` | 同上 | `collect` | 同上 |
| 评论/回复 | `biteblog.interaction` | `interaction.comment` | 同上 | `comment` / `comment_reply` | Post MQ 附加 `commentId`、`parentId`、`parentCommentUserId`、`commentContent` |
| 笔记发布 | `biteblog.post` | `note.published` | `notify.note.published.queue` | `follow_post` | 向 `fans:{authorId}` fanout；大 V（≥50 粉丝 / `feed:bigv` / fans≥500）跳过 |

**未读计数 Redis**（`notify:unread:{userId}`）：写通知 INCR；单条已读 DECR；全部已读 SET `"0"`；DND 写库不 INCR；每 10 分钟与 DB `COUNT(read_status=0 AND is_retracted=0)` 对账。

### 12. 管理员审核日志表 (admin_audit_log)

记录管理员对用户/笔记/评论的审核操作。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 日志 ID |
| admin_id | BIGINT | NOT NULL | 管理员用户 ID |
| action | VARCHAR(20) | NOT NULL | 操作类型（delete/approve/reject） |
| target_type | VARCHAR(20) | NOT NULL | 目标类型（note/comment/user） |
| target_id | BIGINT | NOT NULL | 目标记录 ID |
| reason | VARCHAR(200) | DEFAULT NULL | 操作原因 |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 操作时间 |

**索引：**
- `PRIMARY KEY (id)`
- `INDEX idx_admin (admin_id)` — 加速查询某管理员的操作记录

**使用场景：**
- 管理员删除违规笔记/评论时写入审计日志
- 可追溯：谁在什么时候对什么做了什么操作

## 表关系总览

```
user (1) ──→ (N) follow_relation ──→ (1) user
user (1) ──→ (N) note ──→ (N) note_image
                  │
                  ├──→ (N) note_like ──→ (1) user
                  ├──→ (N) note_favorite ──→ (1) user
                  ├──→ (N) comment ──→ (1) user
                  └──→ (N) user_behavior ──→ (1) user

user (1) ──→ (N) notification (as receiver)
user (1) ──→ (N) notification (as sender)
user (1) ──→ (N) notification_archive (as receiver)
user (1) ──→ (N) notification_preference
user (1) ──→ (N) admin_audit_log (as admin)
```

## Redis 缓存设计

除 MySQL 持久化外，系统使用 Redis 作为缓存层和高频数据存储：

| Key 模式 | 数据结构 | TTL | 用途 | 所属服务 |
|----------|----------|-----|------|----------|
| `cache:user:{id}` | String | 2h | 用户信息缓存 | User |
| `session:{userId}` | String | 24h | 登录会话 | User |
| `follow:{userId}` | Set | - | 关注集合 | User |
| `fans:{userId}` | Set | - | 粉丝集合 | User |
| `post:cache:{postId}` | String | 30min | 笔记详情缓存 | Post |
| `inbox:{userId}` | ZSet | - | Feed 收件箱（score=时间戳） | Feed |
| `exposure:{userId}` | Set | 7-30d | 推荐曝光去重 | Recommend |
| `rank:daily:{yyyy-MM-dd}` | ZSet | - | 日榜热度排行，统计最近 1 天内正常笔记 | Rank |
| `rank:weekly`             | ZSet | - | 周榜热度排行，统计最近 7 天内正常笔记 | Rank |
| `rank:all`                | ZSet | - | 总榜热度排行，统计全部正常笔记        | Rank |
| `location:notes` | GEO | - | 笔记地理位置 | Location |
| `notify:unread:{userId}` | String（纯整数） | 5min | 未读通知计数（INCR/DECR/SET0） | Notify |
| `notify:sender:{userId}` | String | 30min | 发送者昵称二级缓存 | Notify |
| `notify:pref:{userId}` | String（JSON） | 10min | 通知偏好缓存 | Notify |
