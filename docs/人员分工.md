## 非功能性需求总览（摘自需求说明书）

> 以下指标来自《分布式美食探店社区平台_软件需求说明书》第 3.5 节和《概要设计说明书》第 4 节。

| 指标 | 来源 | 要求 |
|------|------|------|
| **列表查询响应时间** | 需求 3.6.3 | 普通列表查询 < 300ms，含复杂条件 < 800ms |
| **推荐页响应时间** | 需求 3.6.3 | 推荐页利用缓存和快速召回，< 1s |
| **笔记发布响应时间** | 需求 3.6.3 | 笔记发布接口应尽快返回（写 MySQL + 返回） |
| **异步事件延迟** | 需求 3.6.3 | Feed 分发、热度更新、推荐刷新、通知发送通过 RabbitMQ 异步，延迟在秒级可接受 |
| **并发度** | 概设 4.2 | 系统支撑一定并发用户量，各服务独立扩展 |
| **数据一致性** | 需求 3.5 | MySQL 保证事务一致，ES/RabbitMQ 保证最终一致（延迟 < 5s） |
| **容错降级** | 需求 3.5 | ES/Redis 不可用时服务降级而非雪崩 |
| **推荐去重** | 概设 6.7 | Redis Set 记录已曝光 postId，避免重复推荐 |
| **大 V 策略** | 需求 2.5 | 粉丝 >= 5000 不 fanout，读 Feed 时实时拉取 |
| **可观测性** | 需求 3.5 | requestId 追踪链路，日志可排查异常 |

### 测试工具说明

**方式一：curl 脚本（轻量，响应时间快速验证）**

适用于响应时间、延迟等指标的快速验证，直接在终端运行。

**方式二：JMeter（正式压测，并发测试）**

适用于并发度、吞吐量。命令行运行：

```powershell
# PowerShell 中使用 & 前缀调用 jmeter.bat
& "D:\from_browser\apache-jmeter-5.6.3\bin\jmeter.bat" -n -t jmeter\<模块>.jmx -l jmeter\<结果>.jtl -e -o jmeter\<报告目录>
```

参数说明：
- `-n`: non-GUI 模式
- `-t`: 测试计划文件
- `-l`: 结果日志文件
- `-e -o`: 生成 HTML 报告

> Windows 下线程数不宜超过 20，循环次数用固定值（不用 -1 无限循环），避免端口耗尽。

**开发任务：**
- [ ] Note/NoteImage/NoteLike/NoteFavorite/Comment 实体 + Mapper
- [ ] PostController：笔记发布（含图片上传 MinIO）、详情查询、删除
- [ ] 笔记全文搜索（Elasticsearch post_index）

- [ ] 评论功能（CommentController：发表、列表分页、支持回复）
- [ ] MySQL 写入 → 同步写 ES post_index → 发送 `note.published` 到 RabbitMQ

**非功能性需求（摘自需求说明书 3.6.3）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| 笔记详情查询 | < 300ms | Redis `post:cache:{postId}` 缓存（TTL 30min） | curl -w |
| ES 全文搜索 | < 800ms | ES post_index 建立 title/content 分词索引 | curl -w |
| 笔记发布 | 尽快返回 | MinIO 异步上传，先返回再写 ES | curl -w |
| 点赞/收藏并发安全 | 无重复数据 | MySQL 唯一约束 `uk_like(note_id, user_id)` | bash 并发循环 + mysql 验证 |
| MySQL → ES 同步 | 最终一致 < 5s | RabbitMQ 异步同步，消费者写 ES | 轮询 ES 验证 |

**性能优化要点：**

1. **缓存策略**：笔记详情缓存在 Redis，key=`post:cache:{postId}`，TTL 30min。缓存命中直接返回，未命中回源 MySQL + 写回缓存
2. **ES 双写**：MySQL 写入成功后发 RabbitMQ 消息，ES 消费者异步写入。MySQL 是数据源，ES 是检索层，允许短暂不一致
3. **点赞幂等**：`note_like` 表 `UNIQUE KEY uk_like(note_id, user_id)`，INSERT ON DUPLICATE KEY 实现点赞/取消切换
4. **计数冗余**：`note.like_count`/`comment_count` 冗余存储，避免 COUNT 查询

### 成员 2 — Feed Service (8083)

**开发任务：**
- [ ] FeedController：关注 Feed 流查询（游标分页）
- [ ] Feed 推模式（Fanout）：普通用户发布笔记 → 粉丝 Redis `inbox:{fanId}` ZSet 写入
- [ ] Feed 拉模式：大 V（粉丝 >= 5000）→ 粉丝读 Feed 时实时拉取合并
- [ ] RabbitMQ 消费 `note.published` 事件 → 执行 Fanout
- [ ] Feign 调用 Post Service 获取笔记详情
- [ ] 已删除笔记过滤

**非功能性需求（摘自需求说明书 3.6.3）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| Feed 流查询 | < 300ms | Redis ZREVRANGEBYSCORE 纯内存，游标分页 | curl -w |
| Fanout 延迟 | 秒级可接受 | RabbitMQ 异步消费 → ZADD inbox | 发布后轮询 Feed |
| 大 V 风暴防护 | 大V 不 fanout | 推拉结合策略 | 检查 inbox 无写入 |
| Feed 空间上限 | 控制内存 | 每个 inbox 保留最近 500 条，ZREMRANGEBYRANK 清理 | redis-cli 验证 |
| 游标分页一致性 | 不丢不重 | ZREVRANGEBYSCORE + score=时间戳 | 连续翻页验证 |

**性能优化要点：**

1. **推拉结合**：核心设计决策。普通用户 fanout 到粉丝 inbox（写扩散），大 V 不 fanout（避免 Fanout 风暴），读 Feed 时实时拉取大 V 内容合并排序
2. **Redis ZSet 存储**：`inbox:{userId}` ZSet，score=笔记发布时间戳。`ZREVRANGEBYSCORE` 实现高效游标分页，O(log N)
3. **空间控制**：每个 inbox 最多保留 500 条，`ZREMRANGEBYRANK inbox:{userId} 0 -501` 定期清理旧数据
4. **Feign Fallback**：Post Service 不可用时返回缓存摘要，不阻塞 Feed 流

### 成员 3 — Recommend Service (8084)

**开发任务：**
- [ ] RecommendController：发现页推荐列表
- [ ] 标签推荐：ES post_index 按标签检索候选集
- [ ] ItemCF 协同过滤：Redis ZSet 物品相似度查询
- [ ] 混合排序：标签推荐(60%) + ItemCF(40%)
- [ ] 冷启动策略：行为数据不足时按热度加权兜底
- [ ] Redis 曝光去重（`exposure:{userId}` Set，TTL 7~30 天）
- [ ] Feign 调用 Post Service、User Service
- [ ] 服务降级：ES 不可用时降级到 Redis 热度榜

**非功能性需求（摘自需求说明书 3.6.3 / 概设 6.7）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| 推荐列表响应 | < 1s | ES 候选召回 + 内存排序 + 曝光去重 | curl -w |
| 曝光去重准确率 | 100% 不重复 | Redis Set SADD/SISMEMBER 实时查 | 连续请求对比 |
| 冷启动可用性 | 新用户有结果 | 热度加权兜底算法 | 新用户访问验证 |
| ES 降级 | 不可用时返回热度榜 | try-catch + Redis fallback | 停 ES 容器验证 |
| 推荐多样性 | 相邻不来自同一作者 | 排序去重策略 | 检查结果列表 |

**性能优化要点：**

1. **Redis 不缓存推荐结果**：推荐是实时计算的（用户行为随时在变），Redis 用于：
   - `exposure:{userId}` Set：曝光去重（状态记录，非缓存）
   - `behavior:{userId}`：用户行为画像缓存（中间数据，TTL 几分钟）
   - `rank:daily:{date}`：冷启动热度榜（复用 Rank Service 数据）
2. **候选召回**：标签候选从 ES 检索，ItemCF 相似度从 Redis ZSet 读取，再在内存中混合排序，避免全量扫描
3. **降级策略**：ES 不可用 → catch 异常 → 直接返回 Redis 热度榜，不报错

**非功能性需求（摘自需求说明书）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| 附近查询响应 | < 300ms | Redis GEORADIUS 纯内存计算 | curl -w |
| GEO 精度 | 距离准确 | Redis GEO 使用 WGS84 坐标系 | 与高德结果对比 |
| 坐标写入延迟 | 秒级 | RabbitMQ 消费 → GEOADD | 发布后轮询附近 |
| 位置授权降级 | 用户拒绝授权时 | 使用默认坐标 + 提示信息 | 前端测试 |
| POI 搜索 | 依赖高德 API | 代理 + 缓存高频查询结果 | curl -w |

**性能优化要点：**

1. **Redis GEO 纯内存**：`GEORADIUS` 是 O(N+log M) 复杂度，纯内存操作，极快
2. **坐标异步写入**：笔记发布时 RabbitMQ 发送坐标事件，Location Service 消费后 GEOADD，不阻塞发布流程
3. **POI 缓存**：高频 POI 查询结果缓存在 Redis，TTL 1h，减少高德 API 调用

- [ ] 监听 RabbitMQ 点赞/收藏/评论事件 → 更新热度分
- [ ] 热度分规则：点赞 +1、收藏 +2、评论 +3、浏览 +0.5

**非功能性需求（摘自需求说明书）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| Top10 查询 | < 100ms | Redis ZREVRANGE 纯内存 | curl -w |
| 热度分更新 | 秒级异步 | RabbitMQ 消费 → ZINCRBY | 点赞后轮询 |
| 定时刷新 | 周期性 | `@Scheduled(fixedRate = 300000)` 5min | 日志观察 |
| 高并发读 | 100+ 并发 | Redis 单线程模型天然保证 | JMeter |
| Top10 白名单 | 无需登录 | Gateway 白名单配置 | 无 Token 访问 |

**性能优化要点：**

1. **Redis ZSet 天然有序**：`ZINCRBY` 更新分数 O(log N)，`ZREVRANGE` 获取 Top N O(log N + M)，极快
2. **按日隔离**：`rank:daily:{date}` 避免历史数据干扰，过期数据自然淘汰
3. **异步更新**：点赞/收藏/评论事件通过 RabbitMQ 异步更新排行，不阻塞业务接口

### 成员 6 — Notify Service (8087)

**开发任务：**
- [ ] WebSocket 连接管理（STOMP over WebSocket）
- [ ] RabbitMQ 消费点赞/评论/关注事件 → 生成通知
- [ ] 通知写入 MySQL notification 表

- [ ] `GET /notify/list`：通知列表（传统分页）
- [ ] `GET /notify/unread-count`：未读计数
- [ ] `POST /notify/read-all` / `POST /notify/{id}/read`：已读操作

**非功能性需求（摘自需求说明书 3.6.3）：**

| 指标 | 要求 | 优化手段 | 测试方式 |
|------|------|----------|----------|
| 通知推送延迟 | 秒级（异步事件） | RabbitMQ 消费 → WebSocket 推送 | 点赞后轮询未读数 |
| WebSocket 连接 | 支持并发连接 | STOMP 会话管理 + 心跳 | 多窗口测试 |
| 断线重连 | 自动重连 | 前端 WebSocket reconnect 逻辑 | 关闭浏览器重开 |
| 通知持久化 | 离线可查 | MySQL notification 表 | 离线产生通知 → 上线查询 |
| 未读计数准确 | 实时 | Redis 缓存 + MySQL 兜底 | 标记已读后验证 |
| 自己不通知自己 | 业务过滤 | receiver_id != sender_id | 自己点赞自己笔记 |

**性能优化要点：**

1. **WebSocket 长连接**：避免轮询，实时推送。STOMP 协议支持心跳检测，断线自动重连
2. **RabbitMQ 缓冲**：通知事件通过 RabbitMQ 异步处理，削峰填谷，避免大量并发写 MySQL
3. **批量消费**：RabbitMQ consumer 可批量拉取消息，批量 INSERT notification，提升写入吞吐

## 联调测试流程

全部服务启动后，按以下流程走一遍完整业务链路：

### 流程一：注册 → 关注 → 发布 → Feed 流

```
1. 组长: 运行 sql/init-data.ps1 创建测试用户
2. 组长: 用户 A 关注用户 B
3. 成员1: 用户 B 发布笔记（含图片 + 坐标）
4. 成员2: 用户 A 刷 Feed 流 → 能看到用户 B 的笔记
5. 成员4: 用户 A 查附近探店 → 能看到用户 B 的笔记坐标
```

### 流程二：互动 → 排行 → 通知

```
1. 成员1: 用户 A 点赞 + 评论用户 B 的笔记
2. 成员5: GET /api/rank/top10 → 热度分更新
3. 成员6: 用户 B 收到点赞 + 评论通知（WebSocket）
4. 成员6: 用户 B 查看通知列表，标记已读
```

### 流程三：推荐 → 搜索

```
1. 成员1: 多个用户发布不同分类笔记
2. 成员3: 用户 A 浏览推荐页 → 返回个性化推荐（含去重）
3. 成员1: ES 关键词搜索 → 返回匹配结果
4. 前端: 搜索图标 → 搜索框 + Top10 笔记展示
```

### 流程四：容错降级验证

```
1. 停掉 ES 服务 → 成员3 推荐接口降级到热度榜（不报错）
2. 停掉 Redis → 成员2 Feed 流降级到直接查 MySQL（不报错）
3. 重启中间件 → 所有服务自动恢复
```

## 协作规范

1. **代码提交**：各成员在自己负责的模块目录下开发，提交前确保 `mvn clean install -DskipTests` 通过
2. **接口对齐**：遵循 [接口规范](接口规范.md)，Controller 返回统一的 `Result<T>` 格式
3. **服务联调**：启动顺序为 `docker compose up` → `init-indices.sh` → `mvn build` → 启动服务 → `init-data.ps1`
4. **Git 分支**：建议各模块在独立分支开发，完成后合并到 main
5. **沟通协调**：模块间通过 Feign 远程调用的接口定义需提前对齐，避免联调时接口不匹配
6. **非功能验证**：每个模块开发完成后，执行上方对应的 curl 测试脚本验证响应时间，并用 JMeter `.jmx` 文件做并发压测，生成 HTML 报告作为验收依据
7. **降级方案**：每个依赖外部服务（ES/Redis/RabbitMQ）的接口必须有 fallback 策略
8. **性能指标来源**：所有响应时间指标来自《需求说明书》第 3.6.3 节，并发度要求来自《概要设计说明书》第 4.2 节
