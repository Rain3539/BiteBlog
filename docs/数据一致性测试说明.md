# BiteBlog 数据一致性测试说明

## 1. 概述

BiteBlog 使用 **MySQL** 为主存储、**Redis** 为缓存/队列、**Elasticsearch** 为搜索引擎、**RabbitMQ** 为异步消息总线。各服务通过 MQ 事件驱动数据同步，形成多条跨服务数据链路。数据一致性测试的目标是验证在这些异构存储之间数据保持一致，不会出现脏数据、丢失、重复或不同步的情况。

### 1.1 系统存储拓扑

| 存储 | 用途 | 涉及服务 |
|------|------|----------|
| MySQL | 主数据存储（10 张表） | 全部服务 |
| Redis | 缓存（ZSet 榜单/inbox/GEO/Pool）、未读数、曝光去重、行为画像 | Feed / Rank / Location / Recommend / Notify |
| Elasticsearch | 全文检索（post_index） | Post / Recommend |
| RabbitMQ | 异步事件总线（note.published / note.deleted / interaction.*） | 全部服务 |

### 1.2 MQ 事件路由总览

```
Post Service 发布/删除/互动
  │
  ├── note.published ──→ FeedEventListener    → Redis feed:inbox (fanout)
  │                   ──→ RankEventListener    → Redis rank:daily/weekly/all
  │                   ──→ LocationEventListener → Redis location:notes (GEO)
  │                   ──→ RecommendEventListener → ES post_index + Redis rank:daily:{date}
  │
  ├── note.deleted  ──→ FeedEventListener    → Redis feed:deleted
  │                   ──→ RankEventListener    → Redis rank:* (ZREM)
  │                   ──→ LocationEventListener → Redis location:notes (ZREM)
  │                   ──→ RecommendEventListener → ES post delete + Redis rank daily remove
  │
  └── interaction.* ──→ RankEventListener    → Redis rank:* (score update)
                       ──→ NotifyEventListener → MySQL notification + Redis unread
                       ──→ RecommendEventListener → Redis rank daily + Redis ItemCF 相似池
```

## 2. 各服务数据一致性测试项

### 2.1 Feed Service

#### 2.1.1 Redis Inbox ↔ MySQL Note 一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| FC-1 | **Fanout 写入一致性** | 普通用户发布笔记后，检查所有粉丝的 `feed:inbox:{fanId}` 中是否包含该 noteId，且 score（时间戳）正确 |
| FC-2 | **大 V 不 Fanout** | 大 V 发布笔记后，验证粉丝 inbox 中不会出现该笔记（仅写入大 V 自身 inbox 作为数据源） |
| FC-3 | **Inbox 预热一致性** | 模拟 inbox 为空（冷启动），触发 `getTimelineFromDb` 降级查询 MySQL 并回填 Redis，验证回填的 noteId 与 MySQL `note` 表中 `status=1` 的记录一致 |
| FC-4 | **删除笔记清理** | 删除笔记后消费 `note.deleted`，检查 `feed:deleted` 集合中是否包含该 noteId，且 fetch feed 时该笔记被过滤 |
| FC-5 | **游标分页无重复/无遗漏** | 连续翻页 5 页（如 size=3），所有 postId 去重后数量等于各页数量之和，无跨页重复 |
| FC-6 | **Redis 不可用降级 MySQL** | 手动停止 Redis 后请求 feed 接口，验证服务不报错，数据直接从 MySQL 查询返回 |

**验证方法**：
- redis-cli 检查 `feed:inbox:{userId}` ZSet members
- mysql 查询 `note` 表对比
- curl 分页结果交叉校验

### 2.2 Rank Service

#### 2.2.1 Redis 榜单缓存 ↔ MySQL Note 一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| RC-1 | **发布入榜一致性** | 发布笔记后消费 `note.published`，检查 `rank:daily:{date}` / `rank:weekly` / `rank:all` 中是否包含该 noteId，且热度分数 = `like*3 + collect*5 + comment*4 + quality + 24/√hours` |
| RC-2 | **互动刷新一致性** | 对笔记执行点赞(+3)、收藏(+5)、评论(+4)后，检查 Redis 中对应榜单的 score 是否同步更新 |
| RC-3 | **删除下榜一致性** | 删除笔记后消费 `note.deleted`，检查三个榜单 ZSet 中该 noteId 是否被移除 |
| RC-4 | **日榜/周榜时间窗口一致性** | 发布超过 1 天的笔记不应出现在 `rank:daily:{today}` 中；超过 7 天的不应在 `rank:weekly` 中出现（由 `shouldJoin` 逻辑控制） |
| RC-5 | **手动重建一致性** | 调用 `POST /rank/rebuild?type=all` 重建全量榜单后，对比 Redis ZSet 与 MySQL note 表按热度公式排序的 TOP N 结果是否一致 |
| RC-6 | **定时重建一致性** | 等待定时任务 `@Scheduled(cron = "0 0/10 * * * ?")` 触发，验证 10 分钟后榜单被自动重建，数据与 MySQL 一致 |
| RC-7 | **榜单容量裁剪** | 持续发布超过 200 条笔记后，验证 Redis ZSet 仅保留 score 最高的 200 条 |
| RC-8 | **分页一致性** | 请求 page=1 和 page=2（如 size=10），验证两页 postId 无交集，排名连续 |
| RC-9 | **MySQL 逻辑删除过滤** | 手动将某笔记 `status` 设为 0，验证该笔记不再出现在分页结果中 |

**验证方法**：
- 发布/互动后 `redis-cli ZRANGE rank:daily:{date} 0 -1 WITHSCORES`
- 对比 MySQL `SELECT * FROM note WHERE status=1 ORDER BY (热度公式) DESC`
- 调用 rebuild 后再次对比

### 2.3 Notify Service

#### 2.3.1 MySQL Notification ↔ Redis 未读数一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| NC-1 | **通知写入一致性** | 互动事件（like/collect/comment）消费后，MySQL `notification` 表中新增一条记录，且 `receiver_id`、`sender_id`、`type`、`biz_id` 与事件消息一致 |
| NC-2 | **未读数 Redis ↔ MySQL 一致** | 写入通知后 Redis `notify:unread:{userId}` 自增 +1，通过 `GET /api/notify/unread-count` 查询结果应与 MySQL `SELECT COUNT(*) FROM notification WHERE receiver_id=? AND read_status=0` 一致 |
| NC-3 | **标记已读后缓存失效** | 调用 `PUT /api/notify/{id}/read` 后，Redis key 被删除，下次查询未读数触发 cache miss → DB COUNT → 回填 |
| NC-4 | **全部已读** | 调用 `PUT /api/notify/read-all` 后，MySQL 中所有 `read_status` 更新为 1，Redis 缓存被 evict |
| NC-5 | **幂等去重（5 分钟窗口）** | 在 5 分钟内重复发送相同 `(receiver_id, sender_id, type, biz_id)` 的 MQ 消息，验证 MySQL 中仅产生 1 条通知记录 |
| NC-6 | **自操作过滤** | 作者给自己的笔记点赞，验证不产生通知（authorId == userId 直接跳过） |
| NC-7 | **分页无跨页重复** | 连续请求 page=1 和 page=2，验证两页 notificationId 无交集（双列排序 `created_at DESC, id DESC` 保证稳定性） |
| NC-8 | **消息可靠性（手动 Ack + DLQ）** | 模拟消费异常（抛 RuntimeException），验证消息流入死信队列 `notify.dead.queue`，主队列不堆积，消息不丢失 |
| NC-9 | **冷热分离归档一致性** | 模拟 30 天前的已读通知，验证定时任务将其从 `notification` 迁移到 `notification_archive`，热表记录数减少等于冷表记录数增加 |
| NC-10 | **未读数降级** | 手动停止 Redis 后请求未读数接口，验证不报错，直接返回 MySQL COUNT 结果 |

**验证方法**：
- MySQL 直接查 `notification` 和 `notification_archive` 表
- redis-cli `GET notify:unread:{userId}` 与 MySQL COUNT 对比
- RabbitMQ 管理台检查 DLQ 消息

### 2.4 Location Service

#### 2.4.1 Redis GEO ↔ MySQL Note 一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| LC-1 | **发布 GEO 写入一致性** | 发布带坐标的笔记后消费 `note.published`，检查 `location:notes` 中是否包含该 noteId，经纬度与 MySQL `note` 表一致 |
| LC-2 | **删除 GEO 清理一致性** | 删除笔记后消费 `note.deleted`，检查 `location:notes` 中该 noteId 是否被移除（ZREM），后续附近查询不应返回该笔记 |
| LC-3 | **GEO 重试机制** | Post Service 事务提交与 Location 消费存在时间差，验证 `addNoteLocation` 的重试逻辑（最多 5 次，间隔 200ms）能在 Post 提交后成功查到 MySQL 中的 note |
| LC-4 | **附近查询结果过滤** | `nearbyMarkers` 返回结果需过 `noteMap` 按 `status=1` 过滤，手动将某笔记 `status` 设为 0 但保留 GEO member，验证该笔记不在附近结果中出现 |
| LC-5 | **POI 缓存一致性** | 首次搜索 "火锅+武汉" 调用高德 API 并写入 Redis，再次搜索直接命中缓存。验证两次返回结果一致，且缓存 TTL=1 小时 |
| LC-6 | **坐标参数校验** | 传入 `longitude=200` 或 `latitude=100`，验证返回 5003 错误码，不会写入脏数据到 GEO |
| LC-7 | **不同半径单调性** | 同一中心点以 1/3/5/10/20km 查询，返回条数应单调非递减 |

**验证方法**：
- redis-cli `GEORADIUS location:notes ...` 返回成员
- MySQL 查 `note` 表 `WHERE longitude IS NOT NULL AND status=1`
- 删除后再次 GEORADIUS 确认移除

### 2.5 Recommend Service

#### 2.5.1 ES ↔ MySQL ↔ Redis 三者一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| RC-1 | **发布 ES 索引一致性** | 发布笔记后消费 `note.published`，检查 ES `post_index` 中是否存在该 `postId`，字段（title/content/shopName/tags 等）与 MySQL `note` 表一致 |
| RC-2 | **删除 ES 清理一致性** | 删除笔记后消费 `note.deleted`，检查 ES `post_index` 中该 `postId` 是否被删除，Redis `rank:daily:{date}` 中该 id 是否被移除 |
| RC-3 | **互动热分更新** | 点赞/收藏/评论后消费 `interaction.*`，检查 Redis `rank:daily:{date}` 中该 noteId 的 score 是否更新（重新计算 `refreshNote`） |
| RC-4 | **Rank 日榜一致性** | `rank:daily:{date}` 由预计算和 MQ 事件写入，验证 score 和 member 与 MySQL 热度公式一致 |
| RC-5 | **预计算定时刷新** | 等待 10 分钟，验证 `scheduledPrecompute()` 自动执行，`rank:daily:{date}` 和 Redis `recommend:itemcf:similar:{postId}` 数据被刷新（而非残留过期数据） |
| RC-6 | **ItemCF 相似池一致性** | 预计算后检查 Redis `recommend:itemcf:similar:{postId}` 中某个 `postId` 的相似候选，score 与非预计算的在线行为扫描结果一致 |
| RC-7 | **行为画像缓存一致性** | 有行为用户的 `behavior:{userId}` Redis 缓存内容应与 MySQL `user_behavior` 表 `WHERE user_id=? ORDER BY created_at DESC` 的记录一致，TTL=300s |
| RC-8 | **曝光去重原子性** | 使用 Lua 脚本 `selectAndReserveUnexposed` 原子预占曝光位，并发请求同一用户的推荐，验证不会出现重复 postId（Redis Set 天然去重 + Lua 原子性） |
| RC-9 | **曝光上报幂等** | 重复调用 `POST /api/recommend/exposures` 上传相同 postId 列表，验证 Redis `exposure:{userId}` SCARD 不增加，推荐结果中这些 postId 被过滤 |
| RC-10 | **冷启动降级链** | 先读 `rank:daily:{date}` → 再降级 MySQL 热门笔记，验证 fallback 正确生效 |
| RC-11 | **ES 不可用降级** | 停止 ES 后请求推荐接口（有 tag 参数），验证降级到 Redis 热榜/冷启动路径，不报错 |
| RC-12 | **分页去重一致性** | 清空曝光集合后连续请求两页推荐，验证两页 postId 无交集 |
| RC-13 | **作者打散一致性** | 推荐列表中相邻 item 的 `authorId` 应尽量不同（diversify 逻辑），验证同作者内容被打散 |
| RC-14 | **边界参数** | `size=999` 被裁剪为 50，曝光上报单次最多 100 条 |

**验证方法**：
- ES `GET post_index/_search`
- redis-cli `ZREVRANGE recommend:itemcf:similar:{postId} 0 -1 WITHSCORES`
- redis-cli `ZRANGE rank:daily:{date} 0 -1 WITHSCORES`
- redis-cli `SMEMBERS exposure:{userId}`
- curl 连续分页交叉校验

### 2.6 Post Service（事件发布源）

#### 2.6.1 MySQL ↔ Redis ↔ ES 一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| PC-1 | **发布事务原子性** | 笔记发布在同一事务中写入 MySQL `note` + `note_image`，事务提交后发 MQ 事件和同步写入作者 inbox。验证任一环节失败时数据不半写入 |
| PC-2 | **详情缓存一致性** | `GET /api/post/{id}` 首次查 MySQL 并写入 Redis `post:cache:{id}`（TTL 30 分钟），再次请求命中缓存。修改或删除笔记后缓存应被清除 |
| PC-3 | **ES 搜索一致性** | Post Service 的 `post_index` ES 索引字段应与 MySQL `note` 表一致。发布后可通过 ES 搜索查到；删除后搜索不返回 |
| PC-4 | **ES 降级** | 停止 ES 后搜索接口不报错，返回空列表（而非 500） |
| PC-5 | **逻辑删除过滤** | 所有查询接口（详情/列表/搜索）不应返回 `status=0` 的笔记 |

## 3. 跨服务端到端一致性测试

### 3.1 发布笔记 → 全链路一致性

| 编号 | 测试项 | 说明 |
|------|--------|------|
| E2E-1 | **发布 → Feed 可见** | 用户 A 发布笔记 → 粉丝 B 的 feed 中出现该笔记。验证链路：MySQL note 表 → MQ note.published → FeedEventListener fanout → Redis inbox → feed API |
| E2E-2 | **发布 → 榜单入榜** | 发布笔记 → `rank:daily:{date}` 中包含该 noteId。验证：MQ note.published → RankEventListener → addInitialScore |
| E2E-3 | **发布 → GEO 可附近搜到** | 发布带坐标笔记 → `location:notes` GEO 中有该 noteId，附近查询可返回 |
| E2E-4 | **发布 → Recommend 可推荐** | 发布笔记 → ES `post_index` 可查到 → Redis `rank:daily:{date}` 有该 noteId → 推荐接口可返回该内容 |
| E2E-5 | **发布 → 所有消费者 Ack** | 发布后检查各队列 ready/unacked 均为 0，确认所有消费者已成功消费 |

### 3.2 删除笔记 → 全链路清理

| 编号 | 测试项 | 说明 |
|------|--------|------|
| E2E-6 | **删除 → Feed 过滤** | 删除笔记 → `feed:deleted` 集合包含该 noteId → feed 接口不再返回该笔记 |
| E2E-7 | **删除 → 榜单移除** | 删除笔记 → `rank:daily/weekly/all` 三个 ZSet 中该 noteId 被移除 |
| E2E-8 | **删除 → GEO 清理** | 删除笔记 → `location:notes` 中该 noteId 被 ZREM |
| E2E-9 | **删除 → Recommend 清理** | 删除笔记 → ES `post_index` 删除 → Redis `rank:daily:{date}` 移除 → 推荐接口不返回 |
| E2E-10 | **删除 → 全链路无残留** | 遍历所有 Redis key / ES index / MySQL 接口，确认删除笔记无残留引用 |

### 3.3 互动 → 多服务联动

| 编号 | 测试项 | 说明 |
|------|--------|------|
| E2E-11 | **点赞 → 榜单 + 通知 + 推荐** | 单次点赞操作同时触发：Rank score +3、Notify 新增通知、Recommend 热门分刷新。验证三条链路均正确消费且 score/count 一致 |
| E2E-12 | **收藏 → 榜单 + 通知 + 推荐** | 同上，收藏权重为 +5 |

| E2E-14 | **取消互动（action=remove）** | 取消点赞/收藏后，Notification 不新增记录（Notify 仅处理 action=add） |

## 5. 测试环境前提

- **Docker 服务**：MySQL 8.0、Redis 7、RabbitMQ 3.12、Elasticsearch（单节点）
- **后端服务**：Gateway(8080) + 7 个业务服务(8081-8087)，全部注册到 Nacos(8848)

## 6. 一致性风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| MQ 消息丢失 | 发布/删除/互动未同步到下游 | 手动 Ack + DLQ，消息持久化 |
| Redis 与 MySQL 不一致 | 榜单/Feed/未读数错误 | Cache-Aside + TTL + 定时重建 |
| ES 与 MySQL 不一致 | 搜索/推荐结果错误 | MQ 驱动增量更新 + precompute 全量刷新 |
| Post 事务未提交时 MQ 已消费 | Location/Recommend 查不到 note | 重试机制（Location 最多 5 次）+ ensureCache 降级 |
| Redis 宕机 | 缓存数据丢失 | 各服务降级到 MySQL，定时任务重建缓存（Rank Scheduled / Recommend Precompute） |
| 并发写入 | 曝光/通知重复 | Redis Lua 原子预占（Recommend）+ 业务层幂等窗口（Notify 5 分钟） |
