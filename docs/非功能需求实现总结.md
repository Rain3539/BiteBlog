# BiteBlog 非功能需求实现总结

> 基于实际代码分析，按性能（高并发）、可用性、可靠性、一致性四个维度，梳理各服务的实现方案。

---

## 一、性能（高并发）

### 1.1 多级缓存体系

| 服务 | 缓存策略 | Redis 数据结构 | TTL | 代码位置 |
|------|------|------|:---:|------|
| Post | **Cache-Aside**：详情读先查 `post:cache:{id}`，miss 查 MySQL 回填 | String (JSON) | 30min | `PostService.getDetail()` |
| Post | 发布/删除时写 ES + 清 Redis 缓存 | — | — | `PostService.publishNote()` / `deleteNote()` |
| Feed | ZSet inbox 缓存 noteId + 时间戳 | ZSet | — | `FeedService.getTimeline()` |
| Rank | ZSet 缓存榜单（noteId + hotScore）| ZSet | — | `RankService.getRankList()` |
| Location | GEO 缓存笔记坐标 | GEO | — | `LocationService.nearbyMarkers()` |
| Location | POI 搜索结果缓存 | String (JSON) | 1h | `LocationService.searchPoi()` |
| Notify | **Cache-Aside**：未读数先查 `notify:unread:{uid}` | String (int) | 5min | `NotifyService.countUnread()` |
| Notify | 昵称二级缓存 `notify:sender:{uid}` | String | 30min | `NotifyService` |
| Notify | 偏好缓存 `notify:pref:{uid}` | String (JSON) | 10min | `NotifyPreferenceService` |
| Recommend | 曝光集合 `exposure:{uid}` | Set | 7d | `RecommendService` |
| Recommend | 行为画像 `behavior:{uid}` | String | 5min | `RecommendService` |
| Recommend | ItemCF 相似池 `recommend:itemcf:similar:{id}` | ZSet | — | `RecommendPrecomputeService` |
| Recommend | 作者信息 `recommend:user:profile:{aid}` | String | 30min | `RecommendService` |
| Recommend | 笔记摘要 `recommend:post:summary:{id}` | String | 30min | `RecommendPrecomputeService` |
| User | 用户信息 `user:cache:{id}` | Hash | — | `UserService` |

### 1.2 读写分离与异步化

| 机制 | 服务 | 说明 |
|------|------|------|
| **推拉结合 Feed** | Feed | 普通用户（粉丝 < 50）推送 fanout 到粉丝 inbox；大V（粉丝 ≥ 50）改为读时拉取，避免写扩散 |
| **ES 独立搜索层** | Post | 全文搜索走 Elasticsearch + IK 分词，与 MySQL 写路径解耦。ES 索引通过 MQ 事件异步更新 |
| **离线预计算 ItemCF** | Recommend | 相似度索引每 10 分钟重建到 Redis ZSet，在线请求只读不计算 |
| **MQ 异步事件驱动** | Post → 全部 | 发布、删除、互动操作通过 RabbitMQ 异步通知 5 个下游服务，主流程不等待，响应不受下游耗时影响 |
| **Feign 异步并发** | Notify | `CompletableFuture` 线程池（核心 5/最大 10）并发查 User Service 昵称，3s 超时 |

### 1.3 数据库优化

| 机制 | 服务 | 代码位置 |
|------|------|------|
| **冗余计数** | Post | 点赞数/收藏数/评论数冗余在 `note` 表，避免每次展示 COUNT |
| **UNIQUE KEY + 原子更新** | Post | `uk_like(note_id, user_id)` 防重复；`SET col = col ± 1` 原子加减防并发覆盖 |
| **子查询批量取封面** | Feed, Post | `SELECT ... (SELECT image_url FROM note_image WHERE note_id=n.id LIMIT 1)` 一条 SQL 搞定，避免 N+1 |
| **双列稳定排序** | Notify | `ORDER BY created_at DESC, id DESC` 防止批量写入时间戳相同时翻页重复 |
| **分页上限** | Rank, Recommend, Notify | size 最大 50，防止一次拉取过多数据 |

### 1.4 前端性能

| 机制 | 说明 |
|------|------|
| 上传与发布分离 | 编辑时可提前上传图片到 MinIO，提交时无需等待 |
| 组件复用不重建 | 同一组件切换笔记时仅刷新数据 |
| 游标翻页 | Feed 和 Recommend 使用 cursor 而非 page，避免新内容插入导致偏移 |
| WebSocket 推送 | 通知和未读数通过 STOMP/WebSocket 实时推送，不轮询 |

## 二、可用性（降级与容错）

### 2.1 多级降级链

```
Feed:
  Redis inbox 不为空 → ZSet 分页返回
  Redis inbox 为空 → MySQL 降级查询（getTimelineFromDb）→ 回填 Redis

Rank:
  Redis ZSet 不为空 → ZREVRANGE WITHSCORES
  Redis ZSet 为空 → ensureCache() 触发 rebuild() 从 MySQL 全量重建

Location nearby:
  Redis GEO 不为空 → GEORADIUS + MySQL 补详情
  Redis GEO 为空 → 定时 rebuildGeo() 从 MySQL 重建

Location POI:
  Redis 缓存命中 → 直接返回
  缓存 miss → 调高德 API → 写回 Redis (TTL 1h)

Post ES 搜索:
  ES 正常 → NativeQuery 搜索
  ES 异常 → catch 后返回空列表（不抛 500）

Recommend:
  冷启动: rank:daily:{date} → MySQL 热门笔记
  个性化: Redis 曝光 → ES 标签召回 → Redis ItemCF → MySQL 在线行为扫描
  补量: 召回不足时从 Rank 日榜 / MySQL 热门补齐
  作者信息: Redis cache → Feign User Service
```

### 2.2 异常安全

| 机制 | 服务 | 代码位置 |
|------|------|------|
| ES 不可用不影响发布 | Post | `publishNote()` 中 `try { esSync.save() } catch ignored` |
| ES 不可用搜索返回空 | Post | `search()` 中 `try { es.search() } catch { return empty }` |
| Redis 不可用时降级 MySQL | Notify | `countUnread()` catch 后 `SELECT COUNT(*)` |
| Redis 曝光读取异常跳过 | Recommend | `exposureRead()` catch 后 warn 日志，跳过曝光过滤继续返回 |
| 高德 API 失败返回 5002 | Location | `searchPoi()` catch 后 throw `POI_SEARCH_FAIL` |

## 三、可靠性

### 3.1 消息不丢失

| 机制 | 服务 | 说明 |
|------|------|------|
| **afterCommit 发送** | Post | `PostEventPublisher` 使用 `TransactionSynchronization.afterCommit()`——MySQL 回滚时 MQ 不发出，杜绝脏事件 |
| **afterCommit 推 WS** | Notify | `NotifyService` 在事务提交后才 Feign 查昵称 + WebSocket 推送，事务回滚不推脏通知 |
| **消息 PERSISTENT** | Post | 所有消息投递 `DeliveryMode.PERSISTENT`，Broker 重启不丢 |
| **Queue durable** | 全部 | 所有 RabbitMQ Queue 声明为 `durable` |
| **手动 Ack + DLQ** | Notify | `ackMode="MANUAL"`，成功 `basicAck`，失败 `basicNack(requeue=false)` → DLX → dead queue |
| **⚠ 已知差距** | Feed, Rank | 消费端用 `AUTO ack` 模式，未配置独立 DLQ。重试耗尽后消息丢弃 |

### 3.2 幂等去重

| 机制 | 服务 | 窗口 | 代码位置 |
|------|------|:---:|------|
| **MySQL UNIQUE KEY** | Post Like/Favorite | 永久 | `uk_like(note_id, user_id)`, `uk_favorite(note_id, user_id)` |
| **5 分钟去重窗口** | Notify | 5min | 写入前 SELECT 同 `receiver+sender+type+bizId` 且 `is_retracted=0` 且 5 分钟内 |
| **Redis Set 天然幂等** | Recommend | 7d | `SADD exposure:{uid}`，重复上报不产生重复记录 |
| **INSERT IGNORE** | Notify | 永久 | 冷归档 `insertIgnore()`，任务重跑不重复 |
| **Redis GEOADD 幂等** | Location | — | 同一 noteId 多次 GEOADD 只会更新坐标 |

### 3.3 软撤回

| 机制 | 服务 | 说明 |
|------|------|------|
| **is_retracted** | Notify | 取消点赞/收藏时 `UPDATE is_retracted=1`，列表过滤；若原未读则 Redis DECR |
| **@TableLogic** | Post | `status=0` = 逻辑删除，MyBatis-Plus 自动追加 `WHERE status=1` |
| **ES status filter** | Post | `NativeQuery.filter(term("status", 1))` 双重过滤已删除笔记 |
| **feed:deleted 惰性过滤** | Feed | 消费 `note.deleted` → SADD `feed:deleted`，timeline 读取时过滤 |

### 3.4 定时自愈

| 机制 | 服务 | 周期 | 说明 |
|------|------|:---:|------|
| **榜单全量重建** | Rank | 每 10 分钟 | 从 MySQL 查询 status=1 笔记，重算热度，覆盖 Redis ZSet |
| **GEO 全量重建** | Location | 每 10 分钟 | 从 MySQL 查带坐标的笔记，重建 `location:notes` |
| **ItemCF 相似池重建** | Recommend | 每 10 分钟 | 重建 Redis ItemCF 相似池 + 补写 Rank 日榜 |
| **未读数对账** | Notify | 每 10 分钟 | 扫描 `notify:unread:*` key，DB>Redis 时用 DB 覆盖 |
| **冷热归档** | Notify | 每天凌晨 3 点 | 30 天前已读通知归档到 `notification_archive` |

### 3.5 跨服务事务补偿

| 机制 | 服务 | 说明 |
|------|------|------|
| **5 次重试等待 Post 提交** | Location | `addNoteLocation()` 查不到 note 时重试 5 次（间隔 200ms），等 Post Service 事务提交 |
| **ensureCache 自愈** | Rank | 查询前检测 ZSet 为空 → 自动从 MySQL 重建，不返回空异常 |

## 四、一致性

### 4.1 MySQL ↔ Redis 缓存一致

| 策略 | 服务 | 写路径 | 读路径 |
|------|------|------|------|
| **Cache-Aside** | Post | 删除笔记 → `DEL post:cache:{id}` | 先查 Redis → miss 查 MySQL → 回填 |
| **Cache-Aside** | Notify | 已读 → SET "0"（不 DEL key）| 先查 Redis → miss 查 DB COUNT → 回填 |
| **单条 DECR** | Notify | 单条已读 → Redis DECR（不 DEL key）| 避免高频逐条已读导致 DB COUNT 回扫 |

### 4.2 MySQL ↔ ES 搜索一致

| 机制 | 说明 |
|------|------|
| **MQ 事件驱动增量同步** | `note.published` → `EsSyncService.savePostToEs()`，`note.deleted` → `EsSyncService.deletePostFromEs()` |
| **延迟 < 1.5s** | 发布后首次轮询 1.5s 内 ES 可搜索到 |
| **双过滤** | MySQL `@TableLogic` + ES `filter(term("status", 1))` 双重保证已删除笔记不可见 |

### 4.3 MySQL ↔ Redis GEO 一致

- 发布 → MQ `note.published` → `LocationEventListener` → `addNoteLocation()` → GEOADD
- 删除 → MQ `note.deleted` → `LocationEventListener` → `removeNoteLocation()` → ZREM
- 附近查询时 `status=1` 过滤，GEO 残留的已删除笔记不会出现在结果中

### 4.4 MySQL ↔ Redis 榜单一致

- 发布/互动事件 → `RankEventListener` → 实时更新 ZSet score
- 手动 `POST /rank/rebuild` 从 MySQL 全量重算
- 定时 `@Scheduled` 每 10 分钟全量刷新，兜底修复任何漂移

### 4.5 Fanout 写入一致（Feed）

- 普通用户发布笔记 → 消费 `note.published` → 遍历 `fans:{authorId}` → 逐个 `ZADD` 到粉丝 inbox
- 所有粉丝 ZSCORE 一致（使用同一发布时间戳）
- 大V 不 fanout（粉丝 ≥ 50），仅写入自身 inbox 供拉取

### 4.6 分页一致性

| 服务 | 机制 |
|------|------|
| Feed, Recommend | 游标（cursor = 时间戳），新内容插入不影响已翻过的页 |
| Notify | `ORDER BY created_at DESC, id DESC` 双列排序，防止同时间戳翻页重复 |
| Rank, Post | ES/page 分页，`PageRequest.of(page-1, size)` |

### 4.7 曝光去重一致性（Recommend）

- **Redis Lua 原子预占**：判断是否已曝光 + 写入曝光集合在 Redis 内原子完成
- 前端幂等上报：重复调用 `/recommend/exposures` → Redis Set 天然去重
- TTL 7 天：避免曝光记录无限增长

## 五、各服务职责汇总

| 服务 | 性能 | 可用性 | 可靠性 | 一致性 |
|------|------|------|------|------|
| **Post** | Cache-Aside, ES搜索, 原子计数器 | ES降级返回空列表 | afterCommit发MQ, PERSISTENT投递, UNIQUE KEY幂等 | @TableLogic+ES双过滤, MQ增量同步ES |
| **Feed** | ZSet inbox, 推拉结合, 子查询封面 | inbox空→MySQL降级回填 | inbox预热兜底, DELETED惰性过滤 | Fanout ZSCORE一致, 游标分页 |
| **Rank** | ZSet 榜单, 热度公式本地算 | ZSet空→ensureCache重建 | 10分钟定时全量重建 | 事件驱动+手动重建+定时兜底 |
| **Location** | GEO纯内存, POI缓存1h | GEO空→定时重建, POI→高德API | 5次重试等Post提交, 10分钟定时重建 | GEO-MySQL事件对齐, status过滤 |
| **Notify** | 未读数Redis, 昵称缓存, CompletableFuture | Redis不可用→MySQL COUNT | 手动Ack+DLQ, 5分钟幂等去重, 10分钟对账 | Cache-Aside,DECR精准维护, 双列排序 |
| **Recommend** | Lua预占, ItemCF预计算, 摘要缓存 | 冷启动→日榜→热门, ES→MySQL降级 | 10分钟预计算自愈, 多级兜底 | Lua原子预占, Redis Set幂等 |
| **User** | Redis Hash缓存 | — | — | — |
